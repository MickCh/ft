# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

`ft` (File Transformer) is a CLI tool for transforming text files: selecting or deleting row/column ranges, reordering rows (sort/tac/shuffle), content filtering (`--grep`), deduplication (`--unique`), case/whitespace transforms (`--upper`/`--lower`/`--trim`), line wrapping (`--wrap`), splitting a row into many (`--split-on`), dropping emptied lines (`--drop-empty`), summaries (`--count`/`--sum`/`--avg`/`--min`/`--max`, optionally per `--group-by`), find/replace within a column range, and in-place editing. Row and column ranges support lists and open-ended bounds; rows also take end-relative (`~N`) ones. Column lists keep their order, so they can permute columns; column ranges can address delimited fields (`--fields`). It is also a learning project for idiomatic Rust — code quality and design matter as much as features.

## Commands

```bash
cargo build                     # build
cargo test                      # run all tests
cargo test test_line_replace    # run a single test by name
cargo clippy --all-targets -- -D warnings   # lint (CI fails on any warning)
cargo fmt                       # format (rustfmt.toml: chain_width 40)
cargo run -- --help             # run the CLI
cargo run -- -R 1-5 -C 2-4 -f foo -r bar input.txt   # example invocation
cargo run -- -F , -C 2-3 data.csv                    # field (delimiter) mode
cargo run -- -R '~10-~1' input.txt                   # end-relative rows (tail)
```

CI (`.github/workflows/rust.yml`) runs `cargo fmt --check`, `cargo clippy --all-targets -- -D warnings`, `cargo build` and `cargo test` on pushes/PRs to `main`.

## Architecture

Library crate (`src/lib.rs`) plus a thin binary (`src/main.rs`). Flow: argv → validated `Config` → composition (`compose::build_processor`) → streaming processor. Only `main.rs` touches the filesystem; the processor works on injected `BufRead`/`Write`, so tests run against in-memory `Cursor`/`Vec<u8>`. Input comes from any number of files and/or stdin (`Config.inputs: Vec<Input>`, `Input::Stdin | Input::File`; never empty, since no argument means stdin).

- `src/main.rs` — three paths: `run_streaming` (file or stdout; reads every input as one stream via `Read::chain`, like `cat`, and rejects an `--output` that aliases *any* input file, which `File::create` would otherwise truncate before the first read), `run_in_place` (`--in-place`, once per input file so each gets its own row numbering — the batch edit; resolves symlinks so the link's target is edited, then writes to a `.<name>.ft-<pid>.tmp` sibling — opened with `create_new` so a planted file or symlink at that name is refused — synced to disk and atomically `rename`d over the input, cleaning up the temp file on any error; `--backup <suffix>` copies the original aside just before the swap) and `report_changes` (`--dry-run`, which writes nothing: `CompareWriter` is a `Write` that compares the result against the original's bytes as they stream by — leftover bytes in the original count as a difference — so each file is reported as "would change" or "unchanged"). `--in-place` requires file inputs (stdin among them is rejected) and conflicts with `--output`. A `BrokenPipe` while writing (e.g. piping into `head`) exits successfully instead of reporting an error.
- `src/cli_args/` — everything between argv and a valid `Config`:
  - `cli.rs` — clap `Command` definition (name/version come from `crate_name!`/`crate_version!`) and range/delimiter parsing. Rows parse into a `RangeSpec` (comma-separated list of parts; each bound is `FromStart`/`FromEnd`, the latter written `~N`); columns parse into a `ColumnList` (comma-separated too, order preserved) and reject `~`. Same-side inverted ranges are rejected here; mixed-bound inversion is only detectable after resolution.
  - `config.rs` — immutable `Config` built via `TryFrom<ArgMatches>`, which validates cross-argument rules (`--replace` requires `--find` and their counts must match; replace/delete are mutually exclusive; delete needs a row/column range or `--grep`; `--ignore-case` needs a pattern; `--in-place` needs file inputs, never stdin). `rows: Option<RangeSpec>`, `cols: Option<ColumnList>`, `sort_key`/`unique_key: Option<ColumnList>` (per-operation column ranges, falling back to `cols`), `output_delimiter: Option<String>`, `reorder: Option<ReorderMode>` (sort/tac/shuffle as one enum, so the flags' mutual exclusivity is structural), `replacements: Vec<Replacement>` (each `--find` pre-paired with its `--replace`; `FindPattern` is a literal or a pre-compiled `Regex`). `rows_or_full()`/`cols_or_full()` supply the `1..=usize::MAX` fallback; `col_span()`/`sort_key_span()`/`unique_key_span()` return a `ColumnSpan` (chars, or fields when `--fields` is set) built by the shared `span_of()`; `has_column_operation()` decides whether a bare `--cols` selects (cuts) columns — an operation given its own key range no longer claims `--cols`.
  - `config_error.rs` — `ConfigError` enum with `Display` for user-facing validation messages.
- `src/compose.rs` — the composition layer: `build_processor(&Config)` assembles the `FileProcessor` (row mode, reordering with its column key, row predicate, the summary reducer, and the transform pipeline in the order delete-cols → bare-select → find/replace pairs → upper → lower → trim → split-on → wrap → drop-empty; the column-scoped transforms run first, wrapping cuts the finished line, and dropping empties sees what the line became). All knowledge of how CLI options map onto engine parts lives here, so the engine modules (`transform`, `predicate`, `file_processor`) never depend on `cli_args`.
- `src/columns.rs` — `ColumnList` (column parts as written *plus* their normalized `RangeSet`: reading honours the written order, writing needs sorted, disjoint parts) and `ColumnSpan` (`Chars(ColumnList)` or `Fields(FieldSpan)`, the latter carrying delimiter, output delimiter, `quoted` and the field list). `field_positions` splits a line into its fields once per resolution — quote-aware under `--quoted`, where a delimiter inside `"…"` does not split and a doubled quote escapes one; a field keeps its quotes, so a projection re-joins into valid CSV. `read_ranges(line)` / `write_ranges(line)` / `delete_ranges(line)` resolve a span to the 1-based char ranges it occupies on a concrete line (as a `Cow`, so char mode borrows and allocates nothing), so field mode reuses the char-indexed helpers. Reading keeps the written order (`-C 3,1,2` permutes) and drops field parts the line does not have; writing normalizes first, which is what makes `-C 2,3` delete exactly what `-C 2-3` does. Delete resolves to swallow one adjacent delimiter, like `cut`; `joiner()` is what rejoins selected fields (`--output-delimiter`, else the input delimiter).
- `src/ranges.rs` — `RangeSet` (normalized, merged, possibly non-contiguous absolute ranges: `contains`, `parts`; also reused by `ColumnList` to normalize columns) and `RangeSpec` (row ranges as written, possibly end-relative). `RangeSpec::is_absolute()` says whether it can resolve without the line count; `resolve(total)` maps every `RangeBound` to an absolute `RangeSet`, dropping empty parts.
- `src/transform.rs` — `LineTransform` trait and its implementations (`SelectColumns`, `DeleteColumns`, `MapColumns` with upper/lower/trim constructors, `ReplaceInColumns`, `ReplaceInColumnsIgnoreCase`, `RegexReplaceInColumns`, `WrapLines`, `DropEmpty`). A transform is **not 1:1**: it returns a `LineOutcome` (`Keep` / `Replace` / `Expand` / `Drop`), so one input line may become several (`--wrap`, `--split-on`) or none (`--drop-empty`). `apply` takes `&mut self`, so a transform may also carry state — `NumberLines` (`--number`) counts the rows it emits, which is why it sits last in the pipeline and conflicts with reordering. `MapColumns` covers every whole-span text mapping (upper/lower/trim/title-case/squeeze) through one constructor each, so those need no new type at all. `Pipeline` owns the transforms and folds a line through them into `Lines` (`One(Cow)` — borrowed while untouched — or `Several(Vec<String>)`, empty once dropped), running every later transform on every expanded line. A new per-line operation is a new transform here, not a new branch in the processing loop. Transforms take an `impl Into<ColumnSpan>` and know nothing about the CLI layer.
- `src/predicate.rs` — `LinePredicate` trait and `GrepPredicate` (`--grep`/`--invert`, scoped to the column span). Selection mode drops non-matching lines; delete mode deletes matching ones.
- `src/reduce.rs` — the third engine abstraction, next to `LineTransform` and `LinePredicate`: a `LineReducer` consumes the processed lines (`accept`) and writes a summary at EOF (`finish`) *instead of* them. `Summarize` implements `--count`/`--sum`/`--avg`/`--min`/`--max` (an `Aggregate` each, with its own `ColumnSpan`), keyed by `--group-by` and printed in first-seen key order (a `Vec` alongside the `HashMap`, so the hash map cannot shuffle the output). A value that does not parse as a number takes no part in a statistic. Reducers see the lines that survive the row range, the predicate, the pipeline and `--unique`, so `--unique --count` counts distinct rows; they conflict with `--delete` and reordering, which would have nothing to act on.
- `src/text.rs` — pure char-indexed helpers (`select_columns` for one range; `select_ranges`, `remove_ranges`, `map_ranges`, `replace_in_ranges` for several; `split_line_terminator`), all resolving a char range to byte offsets in one pass (no per-line `Vec<char>`). `select_ranges` reads the parts in the order given (returning a `Cow`, borrowed for a single part) and joins them; the writing helpers share `rewrite_ranges`, which rebuilds the line left to right and therefore expects ascending, non-overlapping ranges. All UTF-8 edge-case tests live here.
- `src/file_processor.rs` — the engine: `FileProcessor` is a plain struct assembled field by field in `compose` (`RowMode` encodes the three valid row treatments: `Select`, `DeleteSelected`, `EditSelected`). `run` takes `self` by value, because a reducer accumulates into itself and a processor is built fresh per input anyway. `run<R: BufRead, W: Write>`: when `rows.is_absolute()` it streams line by line (`bstr::for_byte_line_with_terminator`) resolving rows against `usize::MAX`; otherwise (end-relative `~N` bounds) it buffers the whole input first, then resolves rows against the real line count. Applies row selection, the row predicate, the transform pipeline and `--unique`, with the terminator split off and re-attached (original terminators, including CRLF, preserved). Because a transform may expand one line into several, `emit` runs per produced line, and the terminator an unterminated line owes is written lazily (`RunState::needs_separator`): only once something actually follows it, so the last line written stays unterminated whether it was reordered, expanded or both.
- `src/constants.rs` — platform-dependent `NEW_LINE` (only used when a reordered/unterminated line needs a terminator).

Key processing concept: transforms stream (each line written immediately), but reordering is *sequence-breaking* — lines inside the row range are buffered in `RunState::reorder_buffer` and flushed before any kept out-of-range line is written and at EOF, so each contiguous run of selected rows reorders in place. `Reorder` unifies the three reordering ops: `Sort` (via `SortSpec`: key `ColumnSpan`, `--numeric` via `NumericKey`/`total_cmp` — unparseable and `NaN` keys sort first, `--reverse` via `cmp::Reverse`, always `sort_by_cached_key` so it stays stable), `Tac` (reverse) and `Shuffle` (rand 0.9). `--unique` keys on the column span of the transformed content via a shared `seen_keys` `HashSet` (first wins; post-sort order with `--sort`). Sort and unique keys use `select_columns` semantics: a span beyond the line yields an empty key, like `cut`. Their spans come from `--sort-key`/`--unique-key` when given, `--cols` otherwise.

All column/row semantics are 1-based inclusive, and column indexing is by `char`, not by byte. End-relative rows count `~1` as the last line. Column lists keep their written order for reading operations (select, sort/unique key, `--grep`) and are normalized for writing ones (delete, case/trim, find/replace).

Tests: unit tests live next to the code; end-to-end CLI tests in `tests/cli.rs` run the real binary via `env!("CARGO_BIN_EXE_ft")` — extend those whenever CLI behavior changes.

## Conventions

- Code, comments, docs, commit messages: English only.
- Design goals for this codebase: SOLID principles, the stairway pattern (depend on traits/abstractions, not concrete types, with dependencies pointing "down the stairs"), and idiomatic Rust (prefer `Result` propagation over `unwrap`, iterators over index loops, borrowed types in signatures).
- Tests live in `#[cfg(test)]` modules next to the code and follow a Given/When/Then comment style.
